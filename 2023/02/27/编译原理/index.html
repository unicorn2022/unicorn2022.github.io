编译原理学习笔记

一、Introduction

  1. 编译器是一个程序，用于将一种语言翻译成另一种语言

    1. 高级语言 => 高级语言/低级语言

  2. 编译器的特点：错误恢复，遇到error不是直接终止程序，而是会返回错误信息

  3. Phases：一个或多个模块，在编译过程中使用不同的抽象“语言”

  4. Interface：描述编译器模块之间交换的信息

三、语法分析 算法

3.1 LL(1)

3.1.1 nullable、First、Follow表的构建

非终止字符 nullable First Follow
S u
B w v, z， y, x
D true y, x z
E true y  x, z
F true x z

Follow()中，一定没有ε

First()中，可能有ε，对应nullable = true。有nullable时可以不写First()中的ε

nullable的计算方法：即该 non-terminal 可以推导出空串

  1.  表中红色：先看能够直接推出ε的规则，将规则左边的nullable项置为true
    1. E => ε，将E的nullable项置为ture
    2. F => ε，将F的nullable项置为ture
  2. 表中蓝色：然后判断剩余规则的右面，如果右面全为nullable，则左边的nullable项置为true
    1. D => EF，将D的nullable项置为ture

First的计算方法：即由该 non-terminal 能够推出的所有字符串中，可能的起始字符为谁

  1. 表中红色：先看规则的右边以 terminal 为起始的规则，则将该 terninal 添加进规则左边 non-terminal 的First()
    1. S => uBDz，将u添加进First(S)
    2. B => w，将w添加进First(B)
    3. E => y，将y添加进First(E)
    4. F => x，将x添加进First(F)
  2. 表中蓝色：然后看剩下的规则A => BC，则First(A) += First(B)；如果B可为空，则First(A) += First(C)
    1. D => EF, E可为空，则First(D) += First(E) ∪ First(F)

Follow的计算方法：即在所有可能的字符串中，可以在该 non-terminal 之后的第一个 terminal

  1. 表中红色：先看规则的右边，如果某个 non-terminal 后面直接跟了一个 terminal，则将该 terminal 添加进该 non-terminal 的Follow()
    1. S => uBDz, D可为空，将z添加进Follow(D)，将z添加进Follow(B)
    2. B => Bv，将v添加进Follow(B)
  2. 表中蓝色：然后看剩下的规则A => BC，则Follow(B) += First(C), Follow(C) += Follow(A)；如果C可为空，则Follow(B) += Follow(A)
    1. S => uBDz，则Follow(B) += First(D)
    2. D => EF, F可为空，则Follow(F) += Follow(D)，Follow(E) += Follow(D)

3.1.2 构建LL(1) Parsing Table

表的作用：给定当前状态、下一个读到的终止字符，可以判定使用哪一条规则继续递归

u z v w x y
S S=>uBDz
B B=>Bv, B=>w
D D=>EF D=>EF D=>EF
E E=> ε E=> ε E=>y
F F=> ε F=>x

对于所有的推导规则A => α：

  1. 表中红色：找出First(α)中包含的所有终止字符t，则T[A,t] = (A=>α)
    1. S => uBDz：First(uBDz) = u，则T[S,u] = (S=>uBDz)
    2. B => Bv：First(Bv) = w，则T[B,w] = (B=>Bv)
    3. B => w：First(w) = w，则T[B,w] = (B=>w)
    4. D => EF：First(EF) = x,y，则T[D,x] = T[D,y] = (D=>EF)
    5. E => y：First(y) = y，则T[E,y] = (E=>y)
    6. F => x：First(x) = x，则T[F,x] = (F=>x)
  2. 表中蓝色：如果α可为空，则找出Follow(A)中包含的所有终止字符t，则T[A,t] = (A=>α)
    1. D => EF：EF可为空，Follow(D) = z，则T[D,z] = (D=>EF)
    2. E =>：Follow(E) = x,z，则T[E,x] = T[E,z] = (E=>)
    3. F =>：Follow(F) = z，则T[F,z] = (F=>)

3.1.3 判断是否为LL(1)文法

核心思路：LL(1) Parsing Table中，某一个项对应多个值

  1. 表中红色：对于所有规则A=>α, A=>β，如果\(First(\alpha)\cap First(\beta) \neq \{\}\)，则不是LL(1)
    1. 不能存在左公因子
  2. 表中蓝色：对于所有的可为空的非终止字符A，如果\(First(A)\cap Follow(A) \ne \{\}\)，则不是LL(1)

3.1.4 将文法修改为LL(1)文法：左递归消除

原来的文法：

  1. \(S \rightarrow S\alpha_1 |...|S\alpha_n\)
  2. \(S \rightarrow \beta_1|...|\beta_m\)

更改为：

  1. \(S \rightarrow \beta_1S' |...| \beta_mS'\)
  2. \(S' \rightarrow \alpha_1S' | ... | \alpha_nS'|\epsilon\)

3.2 LR(0) & SLR(1)

3.2.1 构造LR(0)状态机

保证与起始字符相关的转移规则只有一条，如果不唯一，则新增一个起始字符

  1. 首先，构造起始字符的LR(0)Item，设为状态1
    1. 先将起始字符的唯一一条转移规则，写到状态1里面
      1. S => .X$
    2. 对于状态1中所有的.后面为非终止字符的情况，将终止字符展开，写入状态1里面
      1. 展开X：X => .Ma、X => .bMc、X => .dc、X => .bda
      2. 展开M：M => .d
    3. 如果有一条语句为A => ε，则在Item中的表示为：A => .
  2. 对于所有的状态，从上到下根据.后面的字符，接收该字符，转移到新的状态(没有则无法转移)
    1. 如状态1可以接收b，转移到状态2，对应.向后移一位，即X => b.Mc, X => b.da
    2. 对于新增的状态，类似状态1，进行展开
      1. 如状态2原本是X => b.Mc , X => b.da，但是M为非终止符，需要展开为：M => .d
  3. 直到没有新的状态产生

3.2.2 构造LR(0) Parsing Table

表的内容

  1. sk：将对应的终止字符压栈，然后转移到状态k
  2. gk：将对应的非终止字符压栈，然后转移到状态k
  3. rk：使用第k条规则规约
  4. accept：接受该字符串

表的创建方式：

  1. 对于DFA中的每一个Si 接受终止符t 转移到Sj，令T[i, t] = sj（Input & Shift表）
  2. 对于DFA中的每一个Si 接受非终止符X 转移到Sj，令T[i, X] = gj（Goto表）
  3. 对于每一个可以规约的LR(0) Item，对于其所在所有状态i，每一个终止字符t，令T[i, t] = rk，k为对应规则的编号

3.2.3 构造SLR Parsing Table

首先，求出所有非终止字符的Follow()

  1. Follow(S) = {$}
  2. Follow(E) = {=, $}
  3. Follow(V) = {=, $}

构建SLR(1) Parsing Table：

  1. 对于DFA中的每一个Si 接受终止符t 转移到Sj，令T[i, t] = sj（Input & Shift表）
  2. 对于DFA中的每一个Si 接受非终止符X 转移到Sj，令T[i, X] = gj（Goto表）
  3. 对于每一个可以规约的LR(0) Item (假设为A => α)，对于其所在所有状态i，每一个终止字符t，如果t ∈ Follow(A)，则令T[i, t] = rk，k为对应规则的编号

3.3 LR(1) & LALR(1)

3.3.1 构造LR(1)状态机

保证与起始字符相关的转移规则只有一条，如果不唯一，则新增一个起始字符

  1. 首先，构造起始字符的LR(1)Item，设为状态1
    1. 先将起始字符的唯一一条转移规则，写到状态1里面，并将向前看字符设置为$
      1. S => .X$ ?
    2. 对于状态1中所有的.后面为非终止字符的情况，将非终止字符展开，写入状态1里面，向前看字符为展开非终止字符时，后面的串的First()
      1. 根据S => .X$ ?展开X：
        1. X => .Ma $、X => .bMc $、X => .dc $、X => .bda $
      2. 根据X => .Ma $展开M：
        1. M => .d a
  2. 对于所有的状态，从上到下根据.后面的字符，接收该字符，转移到新的状态，且向前看字符不变(.没有字符则无法转移)
    1. 如状态1可以接收b，转移到状态2，对应.向后移一位
      1. 即X => b.Mc $, X => b.da $
    2. 对于新增的状态，类似状态1，进行展开
      1. 如状态2原本是X => b.Mc $, X => b.da $，但是M为非终止符，需要展开为：M => .d c
  3. 直到没有新的状态产生

3.3.2 构造LR(1) Parsing Table

  1. 对于DFA中的每一个Si 接受终止符t 转移到Sj，令T[i, t] = sj（Input & Shift表）
  2. 对于DFA中的每一个Si 接受非终止符X 转移到Sj，令T[i, X] = gj（Goto表）
  3. 对于每一个可以规约的LR(1) Item，对于其所在所有状态i，每一个向前看字符t，令T[i, t] = rk，k为对应规则的编号

3.3.3 构造LALR(1) Parsing Table

  1. 将LR(1)状态机中，LR(0)部分相同的状态合并，得到LALR(1)状态机
  2. 在根据LALR(1)状态机，仿照构造LR(1)的规则，构造LALR(1)分析表

3.4 将BNF转换为EBNF

3.4.1 {}表示循环

BNF：exp => exp '+' term | term

EBNF：exp => term {'+' term}

3.4.2 []表示选0/1次

BNF：IF => if ( exp ) statement | if ( exp ) statement else statement

EBNF：IF => if ( exp ) statement [else statement]

3.1 上下文无关语法 CFG

  1. 输入：tokens流
  2. 输出：抽象语法树

3.1.2 CFG的表示

  1. 符号：
    1. terminals：终止符
    2. non-terminals：非终止符
  2. 起始符号
  3. 规则：符号 → 符号 符号 ... 符号

  1. 从起始符号开始
  2. 如果存在非终止符号，选择一个规则，用右边的符号替代左边的符号
  3. 重复2，直到只有终止符

3.2 自顶向下分析：Predictive Parsing

LL(k)：

  1. Left-to-right parse：从左到右分析
  2. Leftmost-derivation：最左推导
  3. 1 symbol lookahead：向前看1个符号

3.2.1 递归下降分析法：LL(k)

  1. 可以分析很多CFG，但不是所有的
  2. 关键思想
    1. 每一个非终结符，视为一个递归函数
    2. 每一次推导，视为函数中的一个语句

3.2.2 存在问题：无法解决直接左递归的情况

  1. 预测分析仅适用于：输入中的第一个终止符号提供足够的信息来选择要使用哪种生成的语法
  2. 在解析L时，如果tok=ID，解析器无法确定要使用哪个生成方式

3.2.3 X是否可生成ε：Nullable(X)

  1. \((X:=\epsilon)\)，则\(X\)为\(Nullable\)
  2. \((X:=ABC...)\)，且\(A,B,C...\)均为\(Nullable\)，则\(X\)为\(Nullable\)

3.2.4 X可生成的第一个终结符：First(X)

看规则的左边

  1. 如果\(T\)为终结符，则\(First(T)=\{T\}\)
  2. 如果\(X\)为非终结符，且\((X:=ABC...)\)，则\(First(X)=First(ABC...)\)
    1. \(First(ABC...)=F1 \cup F2 \cup F3 ...\)
    2. \(F1 = Fisrt(A)\)
    3. 如果A为\(Nullable\)，则\(F2=First(B)\)；否则\(F2=\epsilon\)
    4. 如果A、B为\(Nullable\)，则\(F3=First(C)\)；否则\(F3=\epsilon\)

3.2.5 X后面可以跟的第一个终结符：Follow(X)

看规则的右边

  1. 给出语法的第一个规则的左边，默认为起始符号
  2. 对于起始符号，添加一个$进入Follow集合中，$表示输入的结束
  1. 初始化：\(Follow(X):=\{\}\)
  2. 如果\((Y:=s1 X s2)\)，则\(Follow(X)=First(s2)\)
  3. 如果\((Y:=s1Xs2)\)，且\(s2\)为\(Nullable\)，则\(Follow(X)=Follow(Y)\)

  2. 最后一条语句是JUMP、CJUMP

8.2.2 算法

  1. 从头到尾扫描语句序列
  2. 如果发现一个LABEL，则创建一个新的block，之前的block结束
  3. 如果发现一个JUMP、CJUMP，则之前的block结束，下一个block开始

10.1.1 相关定义

Flow-graph terminology：

  1. out-edge：指向后继节点的边，succ[n]表示当前节点的后继节点
  2. in-edge：从前序节点指向当前节点，pred[n]表示当前节点的前序节点

Uses and defs：

  1. use：对变量的访问，即变量出现在赋值语句的右边，use[i]表示当前节点使用到的变量

Liveness：

10.1.7 Static vs Dynamic Liveness

定理：不存在程序H(P, X)，返回如果P(X)可以停机就是true，否则为false

  1. P：任意一个程序
  2. X：P的输入

推论：不存在程序H'(P, L)，返回在P的计算中，L是否可以到达

  1. P：任意一个程序
  2. L：P中的一个label

Dynamic Liveness：

  1. 如果程序的某个执行从节点n到a的一个使用之间没有经过a的任何定义，那么变量a在节点n是动态活跃的

Static Liveness：

  1. 如果存在一条从节点n到a的某个使用的控制流路径，且此路径上没有a的任何定值，那么变量a在节点n是静态活跃的

如果a为动态活跃，则a也为静态活跃。因此编译器通常根据静态活跃信息来进行寄存器分配和其他优化

10.1.8 干涉图 Interference Graphs

  1. 设变量集合为{a, b, c, ...}，寄存器集合为{r1, ..., rk}
  2. 阻止将a和b分配到同一个寄存器的条件称为interference

interference的常见情况：

  1. 活性周期有重合
  2. 如果生成a的指令无法寻址寄存器r1，则a和r1也是interfere的

冲突信息可以用矩阵来表示，也可以用无向图来表示

MOVE指令的特殊处理：

  1. 下列情况中，s和t的内容完全相同，因此不需要添加(s,t)的干涉边
  2. 当然，如果之后还有t的另一个定义，且该时刻s仍然是活跃的，则要创建(s,t)的干涉边

因此，为每一个新定义的值添加冲突边的方法为：

  1. 对于任何定义变量a的非copy指令，以及在该指令处是live-out的变量{b1, ... bj}，添加干涉边(a, b1)，...，(a, bj)
  2. 对于copy指令a←c，以及在该指令处是live-out的变量{b1, ... bj}，对每一个不同于c的bi添加干涉边(a, b1)，...，(a, bj)

  3. 在干涉图上执行着色算法

11.1 通过简化进行着色

  1. 寄存器分配是一个NP-Complete问题
  2. 可以使用一个线性近似算法得到一个较优解

11.1.1 Build 构造

构造冲突图

  1. 利用数据流分析方法，计算在每隔程序点同时活跃的临时变量集合
  2. 由该集合中的每一对临时变量形成一条边。并将这些边加入冲突图中
  3. 对程序中的没一点重复这一过程

11.1.2 Simplify 简化

用一个简单的启发式方法，对图进行着色

  1. 设图\(G\)有一个节点\(m\)，它的相邻节点小于\(K\)个，\(K\)是实际寄存器的个数
  2. 令\(G'=G-\{m\}\)，如果\(G'\)能够使用\(K\)种颜色着色，那么\(G\)也可以
    1. 因为当将m添加到一着色的图\(G'\)时，\(m\)的相邻节点至多使用了\(K-1\)种颜色
    2. 所以总是能找到一种颜色作为\(m\)的颜色

这很自然的导出了一种递归(stack-based or recursive)的图着色算法：

  1. 重复删除\(degree <K\)的节点，并将其压入栈中
  2. 每简化掉一个节点，都会减少其他节点的度数，从而产生更多的简化机会

11.1.3 Spill 溢出

在简化过程的某一点，图\(G\)只包含\(degree\ge K\)的节点

  1. 在图中选择一个节点，并决定在程序执行期间，它存储在memory中
  2. 我们对这个溢出的效果做出乐观的估计，寄希望与这个被溢出的节点将来并不会与留在图中的其他节点发生冲突
  3. 因此可以将这个选中的节点从图中删除，并压入栈中，然后继续进行SImplify操作

11.1.4 Select 选择

将颜色指定给图中的节点

  1. 从一个空的图开始，重复的将栈顶结点添加到图中，重建原来的冲突图
  2. 根据简化阶段的定义，每次往图中添加一个节点时，一定会由一种它可以使用的颜色

当从栈中弹出一个用溢出启发式算法压入栈的潜在溢出节点\(n\)时，并不能保证它是可着色的

  1. 在图中，它的相邻节点可能已经使用了\(K\)中不同的颜色
  2. 在这种情况下，我们就会有一个实际溢出
  3. 此时，我们不知拍任何颜色，而是继续执行Select操作，来识别其他的实际溢出

但是，潜在溢出节点\(n\)的相邻节点，可能使用相同的颜色，导致总颜色种类数\(<K\)

  1. 此时，我们就可以给节点\(n\)着色，并且它不会称为一个实际溢出
  2. 这种方法被称为优化着色 optimisitic coloring

11.1.5 Start Over 重新开始

  1. 如果Select阶段不能为某些节点找到颜色，则必须对程序进行改写，使得在每次使用这些节点之前，将它们从memory中读出，在每次对这些节点赋值之后，将它存回memory中
  2. 这样，一个被溢出的临时变量就会转变为几个具有较小活跃范围的新的临时变量。这些新的临时变量可能会与图中的其它临时变量发生冲突，因此对改写后的程序还要重新进行一次寄存器分配。
  3. 这种处理过程反复迭代，直到没有溢出而简化成功为止

11.2 合并 Coalescing

如果一条MOVE指令对应的两个节点之间不存在边，则可以删除该MOVE指令，并且将两个节点合并为一个节点。

由于合并会导致节点的度数增加，从而可能使图无法进行K着色，因此存在两种合并策略：

11.2.1 合并策略

  1. Briggs：如果节点\(a\)和\(b\)合并产生的节点ab的高度数(\(\ge K\))相邻节点的个数少于\(K\)个，则节点\(a\)和\(b\)可以合并
  2. George：如果对于\(a\)的每一个邻居\(t\)，\(t\)要么与\(b\)有冲突，要么是低度数(\(<K\))的节点，则\(a\)和\(b\)可以合并

11.2.2 带有合并的着色算法

  1. 构造：
    1. 构造冲突图
    2. 将每个节点分为move-related or non-move-related
      1. move-related：该节点是一条MOVE指令的操作数
  2. 简化：
    1. 每次从图中删除低度数的且传递无关的节点
  3. 合并：
    1. 对简化阶段得到的简化图，执行保守的合并
  4. 冻结：
    1. 如果Simplify和Coalesce均不能操作，则找到一个度数较低的传递有关的节点，冻结该节点，然后重新开始Simplify和Coalesce操作
    2. 冻结：放弃对该MOVE指令进行合并的希望，将该节点视为传递无关的
  5. 溢出：
    1. 如果没有低度数的节点，选择一个潜在可能溢出的高度数节点，并将其压入栈
  6. 选择：
    1. 弹出整个栈，并给每个节点着色

11.3 预着色的节点 Precolored Node

  1. 有一些临时变量是预着色的，它们代表的是机器寄存器
  2. 对于每一个由专门用途的真实寄存器，算法应该使用与这些寄存器永久绑定的特殊临时变量
  3. 对于任一给定颜色(机器寄存器)，应该只有一种使用这种颜色的预着色节点
  4. Select和Coalescing操作可以给普通临时变量分配与预着色寄存器相同的颜色，只要它们之间不发生冲突，即：
    1. 一个调用约定的标准寄存器能够在过程中重新用于临时变量
    2. 预着色节点可以通过保守合并算法，与其他非预着色节点合并
  5. 预着色节点有指定的颜色，且不能将其溢出到寄存器

11.3.1 机器寄存器的临时副本

着色算法通过不断执行Simplify、Coalescing、Spill操作，直到只剩下预着色节点，然后Select阶段才能开始向冲突图中加入其他节点，并且对它们进行着色

  1. 预着色节点不能被溢出，因此前端必须尽可能让它们的活跃范围较小
  2. 可以通过生成保存和恢复预着色节点值的MOVE指令来实现这一点
